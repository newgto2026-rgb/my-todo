# 요약
- `CalendarScreen.kt`가 500줄 이상으로 비대해 가독성과 변경 안정성이 떨어진다.
- 캘린더 뷰를 역할별 컴포저블로 분리해 파일 책임을 명확히 한다.
- 분리한 뷰는 `ui/screen`과 `ui/components`에 목적에 맞게 배치한다.
- 동작/디자인은 유지하고, 구조만 개선하는 리팩토링을 수행한다.

# 배경
- 현재 `feature/calendar/impl/ui/screen/CalendarScreen.kt`는 헤더, 월 캘린더 그리드, Agenda 리스트, 아이템 셀, 포맷/헬퍼 로직이 한 파일에 혼재되어 있다.
- 이 구조는 특정 UI 블록만 수정해도 큰 파일 전체를 읽어야 하며, 충돌과 회귀 위험을 키운다.

# 목표
- 화면 엔트리(`CalendarRouteScreen`, `CalendarScreen`)는 조립(Composition) 역할만 담당한다.
- 재사용 가능한 시각 블록을 개별 컴포저블 파일로 분리한다.
- 폴더 규칙을 문맥에 맞게 적용해 뷰 위치를 일관되게 만든다.

# 비목표
- 캘린더 기능(날짜 계산, 필터, 네비게이션) 변경.
- 색상/타이포/간격 등 시각 디자인 개편.
- ViewModel 상태 모델(`CalendarUiState`) 구조 변경.

# 요구사항
## 폴더/배치 규칙
- `ui/screen/`
  - 화면 진입 및 조립 전용 코드만 유지한다.
  - 예: `CalendarRouteScreen`, `CalendarScreen`(컨테이너/오케스트레이션)
- `ui/components/`
  - 화면 하위 재사용 UI 블록을 분리한다.
  - 예: 월 헤더, 주간 요일 헤더, 날짜 셀, Agenda 섹션 헤더, Agenda 아이템
- 헬퍼 함수는 사용 범위에 따라 해당 컴포넌트 파일로 이동하거나, 공통으로 쓰일 경우 `ui/` 하위 별도 util 파일로 분리한다.

## 리팩토링 범위
- `CalendarScreen.kt` 내 아래 단위를 개별 컴포넌트로 분리한다.
  - 월/연도 헤더 및 이전/다음 월 네비게이션
  - 요일 헤더 + 월 날짜 그리드
  - 선택 날짜 Agenda 헤더/카운트 배지
  - Agenda 빈 상태
  - Agenda Todo 아이템
- 상위 화면 파일에서는 상태/콜백 전달과 섹션 배치만 담당한다.

## 코드 품질 규칙
- 사용자 노출 문자열은 리소스 유지(신규 하드코딩 금지).
- 컴포저블 API는 필요한 상태/이벤트만 최소 전달한다.
- 의존성 방향은 유지한다(`feature:calendar:impl` 내부 리팩토링으로 한정).

# 수용 기준
- [ ] `CalendarScreen.kt`의 역할이 화면 조립 중심으로 축소되어, 핵심 레이아웃 흐름을 한 눈에 파악할 수 있다.
- [ ] 하위 뷰 컴포저블이 `feature/calendar/impl/ui/components/`에 목적별 파일로 분리되어 있다.
- [ ] 분리 후 캘린더 화면의 기존 동작(월 이동, 날짜 선택, 일정 클릭 네비게이션)이 동일하게 동작한다.
- [ ] UI 텍스트/리소스 정책 위반(하드코딩 문자열 추가)이 없다.
- [ ] `:feature:calendar:impl:testDebugUnitTest` 및 `:feature:calendar:impl:lintDebug`가 통과한다.

# 검증 범위
- 자동 검증:
  - `./gradlew :feature:calendar:impl:testDebugUnitTest :feature:calendar:impl:lintDebug`
- 수동 검증:
  - 월 이동 버튼 동작
  - 날짜 선택 시 Agenda 목록 동기화
  - Agenda Todo 클릭 시 Todo 편집 진입

# 리스크
- 파일 분리 과정에서 `private` 가시성 변경이 필요해 API 표면이 불필요하게 넓어질 수 있다.
- 컴포저블 파라미터 전달이 과도해지면 오히려 가독성이 악화될 수 있다.
