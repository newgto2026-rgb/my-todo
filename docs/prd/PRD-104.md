# 요약
- 캘린더 화면의 일정 목록에서 Todo를 클릭하면 Todo 화면의 리스트를 거치지 않고 Edit UI(바텀시트)만 즉시 표시한다.
- Edit 종료(닫기/저장/삭제) 후 사용자는 캘린더 컨텍스트로 자연스럽게 복귀한다.
- 라우트 계약은 `feature:todo:api`에 유지하고, 구현은 `feature:todo:impl`에서 처리한다.
- 변경 과정에서 `core:* -> feature:*` 금지 및 기존 모듈 의존 방향을 훼손하지 않는다.

# 배경
- 현재 캘린더 목록 클릭 시 `TodoEditRoute(todoId)`로 이동하지만, Todo 리스트 화면 맥락과 함께 진입되어 사용자가 "편집만 다시 열기" 흐름을 일관되게 느끼기 어렵다.
- 캘린더에서의 사용자 의도는 상세 탐색보다 "즉시 수정"에 가깝다.

# 목표
- 캘린더 진입 시 Edit UI를 최우선으로 노출한다.
- Edit 종료 시 캘린더로 복귀하는 단일 사용자 흐름을 제공한다.
- 기존 Todo 탭(All/Today/Completed)의 동작은 유지한다.

# 비목표
- Todo 에디터의 필드/검증 로직 개편.
- 캘린더 리스트/월간 UI 디자인 변경.
- 모듈 구조(분리 단위) 재설계.

# 요구사항
## 기능 요구사항
- 캘린더 목록 아이템 클릭 시 Todo Edit 전용 진입 모드로 이동한다.
- Edit 전용 진입 모드에서는 리스트/필터/FAB 등 Todo 리스트 UI를 렌더링하지 않는다.
- Edit UI는 기존 Todo 에디터와 동일한 데이터/검증/저장/삭제 동작을 재사용한다.
- Edit UI 닫기/저장/삭제 완료 시 이전 화면(캘린더)로 복귀한다.
- 기존 Todo 내부에서의 일반 편집(리스트에서 항목 클릭)은 기존과 동일하게 동작한다.

## 아키텍처/의존성 요구사항
- 라우트 계약 변경은 `feature:todo:api`에서만 정의한다.
- `feature:calendar:impl`은 `feature:todo:api`까지만 참조하고 `feature:todo:impl`을 참조하지 않는다.
- `core:*` 모듈에 신규 `feature:*` 의존성을 추가하지 않는다.
- `feature:*:entry`의 역할(바인딩/조립) 범위를 넘는 기능 로직을 추가하지 않는다.

# 사용자 시나리오
1. 사용자가 캘린더 탭에서 특정 날짜의 Todo 항목을 클릭한다.
2. 앱이 Todo edit-only 진입 라우트로 이동한다.
3. 사용자는 Edit 바텀시트에서 내용을 수정/저장하거나 닫는다.
4. 작업 완료 후 캘린더 화면으로 돌아와 동일 맥락에서 다음 액션을 진행한다.

# 수용 기준
- [ ] 캘린더 목록 클릭 시 Todo 리스트 본문이 보이지 않고 Edit UI만 표시된다.
- [ ] Edit UI의 저장/삭제/닫기 동작 후 캘린더로 복귀한다.
- [ ] Todo 탭(All/Today/Completed)에서의 기존 편집 동작은 회귀(regression) 없이 유지된다.
- [ ] 모듈 의존성에 신규 역의존(`core:* -> feature:*`) 또는 `calendar:impl -> todo:impl` 참조가 생기지 않는다.
- [ ] `:feature:todo:api`, `:feature:todo:impl`, `:feature:calendar:impl` 빌드/기본 검증이 통과한다.

# 검증 범위
- 수동 검증:
  - 캘린더 -> 항목 클릭 -> edit-only 표시 -> 닫기/저장/삭제 -> 캘린더 복귀
  - Todo All/Today/Completed 탭에서 기존 편집 플로우 동작
- 자동 검증(권장):
  - `./gradlew :feature:todo:api:lintDebug`
  - `./gradlew :feature:todo:impl:testDebugUnitTest :feature:todo:impl:lintDebug`
  - `./gradlew :feature:calendar:impl:testDebugUnitTest :feature:calendar:impl:lintDebug`

# 리스크
- 초기 로딩 타이밍에 따라 `initialEditTodoId` 처리 순서가 꼬이면 edit-only 진입 시 시트가 즉시 열리지 않을 수 있다.
- 네비게이션 백스택 처리(`popBackStack`)와 앱 탭 히스토리 처리 간 상호작용을 점검해야 한다.
